# Native-Initialized Layer4 Draft Selector Distillation

## 目的

验证“微调的 layer4 selector 是否应该从原生 DraftModel layer4 权重初始化，而不是随机初始化”。

对照对象是 full DraftModel selector 产生的 `teacher_scores`。训练 student 时加载 `/mnt/qjhs-sh-lab-01/models/Qwen2.5-3B-Instruct` 的 embedding 和前 4 层作为初始化，使用第 4 层 query-to-doc attention score 拟合 full DraftModel 的 token 分布。

## 数据

- 训练集：`teacher_cache_wikitext103_train_500k`，由 WikiText-103 train 构造 doc/query pair，并用 full DraftModel 生成 teacher score。
- Wiki 验证集：`teacher_cache_wikitext103_val_50k`，当前评估使用前 5k。
- MuSiQue 验证集：`native_initialized_layer4_distill/teacher_cache_musique_val`，由 `data/result_reflect.json` 的 `retrieved_results` 和 main/intermediate query 构造，共 685 条。

## 指标

- `recall_r0p15`：student 选 top 15% doc tokens 后，覆盖 full DraftModel top 15% token 的比例。
- `jaccard_r0p15`：student top 15% 与 full DraftModel top 15% 的 Jaccard。
- epoch 0 是微调前，即原生 layer4 权重初始化后的 baseline。

## 当前观察

- 使用原生 layer4 初始化后，WikiText 分布内验证确实能被 KL 蒸馏拉近 full DraftModel。
- 但同样的更新迁移到 MuSiQue 文本上几乎没有收益，`recall@15` 从 0.5081 到 0.5080，基本持平。
- 这说明问题可能不是随机初始化本身，而是训练数据/目标和真实 RAG 文档分布存在 gap；只在 WikiText 上蒸馏 full DraftModel selector，未必能提升 MuSiQue/RAG 场景。

### 方法命名

1. **FusionRAG-HS：Hidden-Scorer Distillation**
   - 当前 `hidden_head` 路线。
   - Backbone 使用 Qwen2.5-3B 的 embedding + 前 4 层初始化。
   - 额外训练 query-aware hidden scorer head：`score(doc_token, query_pool)`。
   - 训练目标是拟合 full DraftModel selector score，不是标准 causal LM 蒸馏。
   - 当前 best MuSiQue selector-overlap checkpoint：`training_state_epoch003.pt`。

2. **FusionRAG-CD：Causal-Draft Distillation**
   - qjh005 上已 smoke 的 DistillKit 路线，后续改到本机继续。
   - 目标是蒸馏一个真正的小 causal LM / draft model，用 next-token logits KL/CE 等 LM 目标训练。
   - 不显式训练 selector head；推理时读取 student prefill attention 作为 token importance。
   - 当前 qjh005 仅完成 tiny local smoke，真实 Qwen/DraftModel 蒸馏需要迁回本机继续。
